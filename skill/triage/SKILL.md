# Triage

Move GitHub issues in personal non-Wpromote repos through a small state machine. This is not a setup/config skill and does not apply to Wpromote repos.

Every GitHub comment or issue body posted during triage must start with:

```md
> *This was generated by AI during triage.*
```

## Label dependency

GitHub label names, colors, and mappings live in `~/code/scripts/data/github-labels.yml`. Treat that manifest as the source of truth.

If the manifest is absent or unreadable, degrade to **no label automation**:

- State prominently that label automation is unavailable because the manifest dependency is missing.
- Do not guess label names.
- Do not apply partial labels.
- Continue with read-only issue analysis, recommended state, and comment/body drafts the user can review.

## References

- [AGENT-BRIEF.md](references/AGENT-BRIEF.md) — durable AFK-agent issue brief.
- [OUT-OF-SCOPE.md](references/OUT-OF-SCOPE.md) — rejected-enhancement template and `.out-of-scope/` memory rules.

## State model

Track two abstract roles for every triaged issue:

- **Category role** — bug vs enhancement.
- **State role** — awaiting maintainer triage, awaiting reporter info, ready for AFK agent, ready for human, or rejected.

The concrete GitHub labels for those roles come only from `~/code/scripts/data/github-labels.yml`.

Normal transitions:

1. Unlabeled issue -> awaiting maintainer triage.
2. Awaiting maintainer triage -> awaiting reporter info, ready for AFK agent, ready for human, or rejected.
3. Awaiting reporter info -> awaiting maintainer triage once the reporter replies.

Maintainers can override. If an issue appears to have conflicting state roles, stop and ask before changing anything.

## Invocation

Interpret `/triage` or natural-language triage requests, for example:

- "Show me issues needing attention."
- "Let's triage #42."
- "Move #42 to ready for an agent."
- "What's ready for agents to pick up?"

## Show what needs attention

Query GitHub issues and present three buckets, oldest first:

1. Unlabeled issues.
2. Issues awaiting maintainer triage.
3. Issues awaiting reporter info where the reporter has replied since the last triage note.

Show counts and a one-line summary per issue. Let the maintainer pick.

## Triage a specific issue

1. **Gather context.** Read the full issue: body, comments, labels, reporter, and dates. Parse prior triage notes. Explore relevant code using domain glossary terms and respect ADRs. Read target-repo `.out-of-scope/*.md` and surface similar prior rejections.
2. **Recommend.** Give category and state recommendations with reasoning plus a concise codebase summary. Wait for maintainer direction.
3. **Reproduce bugs.** Before grilling, attempt reproduction from reporter steps by tracing code and running focused tests/commands when appropriate. Report success, failure, or insufficient detail.
4. **Grill if needed.** If the issue needs fleshing out, run `grill-with-docs` before drafting final issue text.
5. **Apply the outcome.** Confirm intended GitHub mutations before acting: label changes, comments, issue body edits, or closing.

Outcome handling:

- Ready for AFK agent -> post an agent brief comment using [AGENT-BRIEF.md](references/AGENT-BRIEF.md).
- Ready for human -> use the same structure, but state why it cannot be delegated.
- Awaiting reporter info -> post triage notes with specific questions.
- Rejected bug -> post a polite explanation and close.
- Rejected enhancement -> write target-repo `.out-of-scope/` memory only through Aragorn, link it from a comment, then close.
- Awaiting maintainer triage -> apply the state role when label automation is available; optional comment if useful.

Per-project `.out-of-scope/` memory lives in the target repo being triaged. Writes are Aragorn-only and user-visible.

## Quick state override

If the maintainer says to move an issue to a state, trust them. Confirm the concrete changes, then apply them. Skip grilling. If moving to ready for AFK agent without a grilling session, ask whether they want an agent brief first.

## Needs-info template

```md
> *This was generated by AI during triage.*

## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

Capture resolved facts under "established so far" so the work is not lost. Questions must be specific and actionable.

## Resuming previous triage

Read prior triage notes, check whether the reporter answered outstanding questions, and present the updated picture before continuing. Do not re-ask resolved questions.
